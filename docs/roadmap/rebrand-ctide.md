## 0. TL;DR

- **Naming map**: workspace/repo `cmux-terminal-ide`, binary + brew formula `ctide`,
  crate stem `ctide-*` (1:1 with the design-plan `cide-*` names). See §1.

  syntactically valid crate names and (per crates.io API 404s on 2026-06-12)
  *appear* unregistered; `breezy-bays-labs/homebrew-tap` does **not** yet exist.

Dependency rule (CI-enforced, design-plan §2) is name-agnostic and carries over verbatim:
`ctide-core` depends on nothing in the workspace; adapters depend on core (+ `ctide-json`
where they serialize); the binary depends on everything; nothing depends on the binary.

**Pattern precedent — crap4rs.** The owner's own published workspace
(`/Users/cmbays/github/crap4rs`) is the exact shape: a `[workspace]` root with member
crates `crap-core` (the library hexagon) + `crap4rs` (the binary) + siblings
(`crap-examples`, `crap4ts`). `ctide-core` + `ctide` is the same split — repo-name ≠
crate-stem ≠ binary-name is normal and already proven in-house (also ripgrep/rg,
prior-decisions §1).

### 1.3 What does NOT change name

- **The shell dogfood** keeps `cide-*` / `cwd*` until deletion (not rename — §3).
- **Recipe/vertical type names**: `base`, `dbt`, `rust-dev` are data values, not brand.
- **cmux** and every third-party tool (helix, yazi, lazygit, harlequin, watchexec,
  cute-dbt) — external, untouched.
- **`.cmux/` directory** — cmux's namespace, not ours.

## 2. Repo strategy: rename vs fresh repo

### 2.1 Option A — RENAME `cmux-workspace-dbt` → `cmux-terminal-ide` (RECOMMENDED)

**Pros**
1. **The golden-master "permit" lives here.** The POSIX emitted-command suite
   (~120 assertions in `tests/run.sh`; design-plan §8.2, prior-decisions §1) is *the*
   strangler safety net: a Rust verb may not replace its shell twin until its
   `FakeMux` op log diffs clean against that suite. The suite is in `tests/` in *this*
   repo. A fresh repo either copies it (losing provenance/history) or leaves it behind
   (losing the permit).
2. **Strangler coexistence requires one tree.** Design-plan §9 is explicit: each
   `bin/cide-*` script grows a 3-line exec preamble so the binary takes over
   verb-by-verb, with `CTIDE_SHELL=1` as instant rollback. Old shell and new Rust must
   live side-by-side, in the same repo, through R1–R5. Two repos make per-verb cutover
   and rollback structurally impossible.
3. **The approved vision + design + 18-file evidence corpus are already here**
   (`docs/vision/`). Renaming keeps `git blame`, PR #23 history, and every internal
   doc cross-link intact. A fresh repo orphans the merge history of the decision that
   authorized the work.
4. **Full git history + dogfood scar tissue.** The "quirk vault" (design-plan §4) is
   *paid-for-live* cmux facts captured in shell + fixtures in this repo's history. That
   history is design input, not trivia.
5. **GitHub auto-redirects** old `…/cmux-workspace-dbt` URLs (web + git remote) to the
   new name, so external links/clones don't hard-break on day one.
6. **Fidelity snapshots** (`fidelity/snapshots/0.64.x/`) — the versioned cmux capture
   corpus risk #1 mitigation depends on — stay attached to their history.

**Cons / costs**
1. **Legacy `dbt-workspace` identity in the URL stem is gone but its history remains.**
   The repo's whole Phase-0 past is dbt-shell-centric; a reader browsing history sees a
   dbt workspace tool becoming a general IDE. (This is honest, not harmful — it *is* the
   story.)
2. **Redirect is not forever / not load-bearing.** GitHub redirects break if anyone
   later creates a *new* repo at the old name (it cannot be re-used while the redirect
   is live, but a redirect is silently dropped if the old name is re-occupied by a
   transfer). Treat redirects as a courtesy, not infrastructure — update every
   first-party reference explicitly (§4.3).
3. **In-repo string churn.** `cmux-workspace-dbt` / `DBT_WS_*` / `cwd` appear across
   `install.sh`, `README.md`, `justfile`, `lefthook.yml`, `profiles/*.env`,
   `lib/*.sh`, `bin/*`, `tests/`, `fidelity/` (verified: ~45 files match). Most of
   these are shell-dogfood files that R1–R5 *delete* anyway, so the churn is bounded
   and largely self-resolving — only the small set that survives the strangler
   (`README.md`, `justfile`, CI badge, `LICENSE` header) needs deliberate editing.

### 2.2 Option B — fresh `cmux-terminal-ide` repo

**Pros**
1. **Clean public identity from commit one** — no dbt-workspace legacy in the history.
2. **dbt-workspace shell legacy "left behind"** in an archived repo; the new repo is
   pure Rust from the start.
3. Slightly tidier first impression for a future open-source reader.

**Cons (decisive)**
1. **Kills the strangler.** No coexistence tree → no per-verb cutover, no
   `CTIDE_SHELL=1` rollback, no incremental shipping. The design plan's entire R1–R5
   risk posture (risk #8: "two-generation migration limbo") collapses into a big-bang
   rewrite — *the exact thing prior-decisions §1 explicitly rejected* ("Big-bang
   rewrite; parallel greenfield repo").
2. **Orphans the golden-master permit.** Copying the suite loses its provenance and the
   captured-from-real fixture history (fixture-provenance rule, prior-decisions §17).
3. **Splits the docs corpus** from the code it documents, or forces a doc migration
   that re-introduces stale cross-links.
4. **prior-decisions §1 already ruled on this**: "repo rename `cmux-workspace-dbt` →
   `cmux-ide` is colocated, executed at Cargo-scaffolding time, not mid-shape … parallel
   greenfield repo [is rejected]." A fresh repo re-litigates a settled decision.

### 2.3 RECOMMENDATION

**Rename in place (Option A).** It is the only option compatible with the locked
strangler-fig migration and the golden-master permit, both of which the design plan
treats as non-negotiable. The fresh-repo "clean identity" benefit is cosmetic and is
already rejected in prior-decisions §1. The repo only goes *public* at the brew-tap
milestone (design-plan §10, post-v1); the dbt-workspace history is private and harmless
until then, and is genuine provenance, not baggage.

> **Note on the stale `cmux-ide/` label.** The design plan still literally says
> `cmux-ide/` in §2 ("repo, post-rename") — that is a **pre-rebrand artifact**
> predating the 2026-06-09 decision. The owner's rebrand supersedes it with
> `cmux-terminal-ide`. **The authoritative repo/workspace name is
> `cmux-terminal-ide`** — a builder mirroring §2's directory layout must NOT use
> `cmux-ide/` as the dir name. The *strategy* (rename in place at Cargo-scaffolding
> time) is unchanged; only the destination name moved from `cmux-ide` to
> `cmux-terminal-ide`.

## 3. Born-new `ctide` vs `cide-*` shell during the strangler

The rebrand changes *nothing* about the strangler mechanism (design-plan §9) — it only
fixes the names of the new generation. The two generations stay name-distinct on purpose,
which makes "which generation owns this verb?" trivially greppable.

### 3.1 Born-new `ctide` (Rust, named `ctide` from crate one)

- The entire Cargo workspace: `ctide-core`, `ctide-json`, `ctide-mux-cmux`,
  `ctide-adapters`, `ctide-dbt`, `ctide-place-macos`, `ctide-testkit`, and the `ctide`
  binary.
- The binary's verbs: `ctide doctor`, `ctide state migrate`, `ctide theme`,
  `ctide agent ls`, `ctide statusline` (R1); `ctide run` / `run wrap`, `ctide jump`,
  `ctide open`, … (R2+) — every verb in design-plan §9's phase table, `cide`→`ctide`.
- `ctide.toml` / `ctide.local.toml`, `~/.local/state/ctide/`, `CTIDE_SHELL`, the brew
  formula `ctide`, the tap `homebrew-tap`.
- New-capability verbs that **never existed in shell** (the runner `ctide run wrap`,
  `ctide sync`, `ctide review`, `ctide policy`, `ctide replace`, `ctide setup`) are born
  `ctide`-only — design-plan §9 rule (4): no new shell features after R1.

### 3.2 Stays `cide-*` / `cwd*` shell (retired, not renamed)

- `bin/cide-space`, `bin/cide-place`, `bin/cide-agent`, `bin/cide-jump`, `bin/cide-open`,
  `bin/cide-md-open`, `bin/cide-set-role`, `bin/cide-regen`, `bin/cide-theme`,
  `bin/cide-yazi`, plus `bin/cwd`, `cwd-focus`, `cwd-route` and the wrap launchers
  (`hx-wrap`, `yazi-wrap`, `hq-wrap`, `btop-wrap`, `git-glance*`).
- These keep their current names. Each gets the 3-line exec preamble:
  `command -v ctide >/dev/null && [ -z "$CTIDE_SHELL" ] && exec ctide <verb> "$@"`
  (design-plan §9, with `CIDE_SHELL`→`CTIDE_SHELL`). The shell *body* is deleted only
  when the Rust twin passes golden master, per the phase table.
- **Wrapper-launcher scripts stay shell forever** (design-plan §9, R5 note + risk #6):
  `hx-wrap` etc. are *adapters' launchers*, not verbs — they call `ctide set-role` but
  remain shell. They are not "retired"; they are the permanent shell edge of adapters.
- The POSIX golden-master suite (~120 assertions) stays as the behavioral oracle until
  R5, where it converts to cucumber `.feature` files (design-plan §9, R5).

### 3.3 Why retire-not-rename is correct

Renaming the shell to `ctide-*` would (a) imply the shell is the new product (it is the
*old* one being strangled), (b) double the rename churn for files slated for deletion,
and (c) break the clean grep boundary (`cide-*` = dying generation, `ctide`/`ctide-*` =
living generation) that makes the strangler auditable. The owner's decision (retire,
don't rename) is the low-churn, high-clarity path.

## 4. When the rename happens — R0, and what it gates

### 4.1 The rename is a discrete R0 step before R1

Design-plan §9's phase table starts at R1 ("foundations"). The repo rename is the
implicit **"Cargo-scaffolding time"** that prior-decisions §1 and design-plan §2 both
name ("repo, post-rename (executed at Cargo-scaffolding time)"). Make it an explicit
**R0** so it is tracked and gated, not assumed:

> **R0 — rebrand + scaffold (the one-time setup before any Rust verb).**
> Ships: GitHub repo rename (`cmux-terminal-ide`), remote/URL fixups, ops + memory path
> fixups (§2.4), the brew tap repo `homebrew-tap` created (empty), the `[workspace]`
> `Cargo.toml` skeleton with the 8 `ctide-*` crate dirs stubbed, the crap4rs-derived CI
> scaffold (release-plz, deny.toml, lefthook, rust-toolchain pin, cucumber harness),
> and the `cargo-dist` config for the per-arch darwin + musl-linux artifacts
> (design-plan §10). Retires: nothing (read-only setup). Rollback: trivial — no verb has
> moved yet; `CTIDE_SHELL` is irrelevant because `ctide` does nothing yet.

### 4.2 What R0 gates

- **R1 cannot start** until the workspace name + crate names are fixed, because R1 lands
  the actual crates (`ctide-core`, `ctide-json`, `ctide-mux-cmux`, the binary skeleton).
  Naming churn after R1 would mean renaming compiled crate dependencies, published-crate
  intent, the brew formula, and the `CTIDE_*`/state-dir constants mid-flight.
- **The crates.io reservation** (publishing `ctide-testkit` at R3/R5, open Q#6) and the
  **brew tap publish** (design-plan §10, post-v1) both inherit the names fixed at R0.
- **The `CTIDE_SHELL` rollback contract** that every R1+ exec-preamble depends on is
  named at R0.

### 4.3 What R0 does NOT gate / can lag

- The **local working-dir rename** and the **public repo visibility flip** are
  independent of R0 and can lag — the repo stays private until the brew tap goes public
  (post-v1). Recommendation: rename the GitHub repo at R0 (cheap, redirect-covered),
  but defer the *local dir* rename to avoid the memory-path disruption (§2.4 step 6)
  until it is convenient — the local path is cosmetic.

## 5. Identifier validity + availability (light web-check)

> Verified 2026-06-12. crates.io reads via its HTTP API are cache-prone and US-only;
> treat them as indicative, re-confirm with the authoritative tools at scaffolding time.

### 5.1 Cargo / crates.io naming rules (apply to every `ctide-*`)

- Allowed chars: alphanumeric + `-` + `_`; non-empty
  (<https://doc.rust-lang.org/cargo/reference/manifest.html>).
- crates.io publish constraints: **ASCII only, max 64 chars, no reserved names, no
  Windows special names (`nul`, …)**. `ctide` (5), `ctide-core` (10), longest
  `ctide-place-macos` (17) — all well under 64, all ASCII, none reserved.
- Hyphen→underscore: crate name `ctide-core` → lib import name `ctide_core` (standard;
  the lib-name column in §1.2 reflects this).
- `cargo new`/`init` additionally require a valid Rust identifier — satisfied (no
  leading digit, no keyword). The binary target `ctide` is a clean identifier.
- **Conclusion: every proposed identifier is syntactically valid for crates.io.**

### 5.3 Homebrew rules (confirmed)

- Third-party taps are explicitly permitted; formula names in a private tap are
  arbitrary (the owner sets `ctide`).
- Tap repo *recommended* to start with `homebrew-` so users can write the short form:
  repo `breezy-bays-labs/homebrew-tap` → `brew install breezy-bays-labs/tap/ctide`
  (<https://docs.brew.sh/How-to-Create-and-Maintain-a-Tap>,
  <https://docs.brew.sh/Acceptable-Formulae>).
- `cargo-dist` generates the per-arch formula (`on_arm`/`on_intel`, no universal lipo)
  and can push it to the tap — design-plan §10. (Note: crap4rs ships release-plz +
  deny.toml + lefthook + rust-toolchain pin + cucumber, but **not** cargo-dist —
  cargo-dist for brew distribution is net-new for ctide; the rest is reuse.)
